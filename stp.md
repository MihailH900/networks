# STP и RSTP

Основная настройка выполняется на MikroTik через WebFig. Для Rapier приведена отдельная ветка AlliedWare CLI.

Перед началом выполните подготовку из [`00-basic-setup`](../00-basic-setup/).

Связанные методички:

* [`02-vlan`](../02-vlan/) — VLAN и trunk;
* [`03-link-aggregation`](../03-link-aggregation/) — LACP и bonding.

Обозначения:

* `<BRIDGE>` — bridge, на котором работает STP/RSTP;
* `<CLIENT_PORT>` — порт конечного устройства;
* `<SWITCH_PORT>` — соединение с другим коммутатором;
* `<ROOT_PRIORITY>` — приоритет основного Root Bridge;
* `<BACKUP_PRIORITY>` — приоритет резервного Root Bridge;
* `<PATH_COST>` — стоимость пути через порт.

<a id="mode-selection"></a>

## 1. Выбор STP или RSTP

Для новых конфигураций используйте:

```text
Protocol Mode: rstp
```

Классический STP используйте только по прямому требованию задания:

```text
Protocol Mode: stp
```

| Режим  | Назначение                       |
| ------ | -------------------------------- |
| `stp`  | классический IEEE 802.1D         |
| `rstp` | быстрая сходимость IEEE 802.1w   |
| `mstp` | отдельные деревья для групп VLAN |
| `none` | защита от L2-петель отключена    |

В обычном экзаменационном задании достаточно RSTP.

Все коммутаторы одной резервной L2-топологии должны участвовать в STP/RSTP.

### Основное правило

RSTP нужен, когда между коммутаторами существуют несколько L2-путей:

```text
       SW1
      /   \
    SW2───SW3
```

Один из путей будет временно переведён в состояние `Discarding`. При отказе активного пути резервный порт перейдёт в `Forwarding`.

### Безопасный порядок подключения

Не замыкайте кольцо до включения RSTP.

Правильный порядок:

1. соединить устройства цепочкой;
2. создать bridge на каждом коммутаторе;
3. включить RSTP;
4. настроить межкоммутаторные порты;
5. только после этого подключить последний кабель и замкнуть кольцо.

<a id="mikrotik-ring"></a>

## 2. Кольцо из трёх MikroTik

### 2.1. Физическая схема

Используем три MikroTik:

```text
                 SW1
           ether5   ether6
              /       \
             /         \
      ether5             ether5
          SW2────────────SW3
       ether6          ether6
```

Таблица соединений:

| Кабель | Первый конец | Второй конец |
| ------ | ------------ | ------------ |
| 1      | SW1 `ether5` | SW2 `ether5` |
| 2      | SW1 `ether6` | SW3 `ether5` |
| 3      | SW2 `ether6` | SW3 `ether6` |

Клиентские порты:

| Устройство | Порт клиента |
| ---------- | ------------ |
| SW1        | `ether2`     |
| SW2        | `ether2`     |
| SW3        | `ether2`     |

Для управления рекомендуется использовать отдельный порт, например `ether10`.

### 2.2. Предварительное подключение

Сначала подключите только:

```text
SW1 ether5 ↔ SW2 ether5
SW1 ether6 ↔ SW3 ether5
```

Кабель:

```text
SW2 ether6 ↔ SW3 ether6
```

подключите после настройки RSTP на всех трёх устройствах.

### 2.3. Использовать существующий bridge или создать новый

Если на устройстве уже существует VLAN-aware bridge, который должен участвовать в резервной топологии, используйте его.

Не создавайте второй bridge поверх существующей VLAN-конфигурации.

Если конфигурация начинается с нуля, на каждом MikroTik откройте:

> **Bridge → Bridge → Add New**

Укажите:

| Поле           | Значение  |
| -------------- | --------- |
| Name           | `br-rstp` |
| Protocol Mode  | `rstp`    |
| VLAN Filtering | `no`      |

Для VLAN-сети значение `VLAN Filtering` настраивается по методичке [`02-vlan`](../02-vlan/).

### 2.4. Добавить порты на SW1

Откройте:

> **Bridge → Ports → Add New**

Добавьте `ether5`:

| Поле           | Значение  |
| -------------- | --------- |
| Interface      | `ether5`  |
| Bridge         | `br-rstp` |
| Edge           | `no`      |
| Point To Point | `yes`     |
| BPDU Guard     | `no`      |

Повторите для `ether6`.

Добавьте клиентский `ether2`:

| Поле       | Значение  |
| ---------- | --------- |
| Interface  | `ether2`  |
| Bridge     | `br-rstp` |
| Edge       | `yes`     |
| BPDU Guard | `yes`     |

### 2.5. Добавить порты на SW2

В `br-rstp` добавьте:

```text
ether5 — соединение с SW1
ether6 — соединение с SW3
ether2 — клиент
```

Для `ether5` и `ether6`:

```text
Edge: no
Point To Point: yes
BPDU Guard: no
```

Для `ether2`:

```text
Edge: yes
BPDU Guard: yes
```

### 2.6. Добавить порты на SW3

В `br-rstp` добавьте:

```text
ether5 — соединение с SW1
ether6 — соединение с SW2
ether2 — клиент
```

Параметры такие же, как на SW2.

### 2.7. Замкнуть кольцо

После настройки RSTP на всех устройствах подключите:

```text
SW2 ether6 ↔ SW3 ether6
```

RSTP должен определить избыточный путь и перевести один из портов в роль `Alternate` и состояние `Discarding`.

<a id="root-bridge"></a>

## 3. Настройка Root Bridge

Root Bridge выбирается по наименьшему Bridge ID:

```text
Bridge Priority + MAC-адрес
```

Чтобы выбор не зависел от случайного MAC, задайте приоритеты вручную.

Рекомендуется использовать значения с шагом `4096`.

| Коммутатор | Роль           | Priority |
| ---------- | -------------- | -------: |
| SW1        | основной Root  |   `4096` |
| SW2        | резервный Root |   `8192` |
| SW3        | обычный bridge |  `12288` |

### 3.1. Основной Root — SW1

На SW1 откройте:

> **Bridge → Bridge → br-rstp**

Установите:

```text
Priority: 4096
```

В некоторых версиях WebFig значение можно указать шестнадцатерично:

```text
0x1000
```

### 3.2. Резервный Root — SW2

На SW2:

```text
Priority: 8192
```

или:

```text
0x2000
```

### 3.3. Приоритет SW3

На SW3:

```text
Priority: 12288
```

или:

```text
0x3000
```

### 3.4. Допустимые значения

Используйте:

```text
0
4096
8192
12288
16384
20480
24576
28672
32768
...
```

Чем меньше число, тем выше приоритет устройства при выборе Root Bridge.

Не задавайте одинаковый приоритет всем коммутаторам, если требуется предсказуемый Root.

<a id="switch-ports"></a>

## 4. Настройка межкоммутаторных портов

Для каждого соединения switch-to-switch откройте:

> **Bridge → Ports → `<SWITCH_PORT>`**

Установите:

| Поле           | Значение                                |
| -------------- | --------------------------------------- |
| Edge           | `no`                                    |
| Point To Point | `yes` или `auto`                        |
| BPDU Guard     | `no`                                    |
| Path Cost      | `auto`, если путь не требуется изменить |
| Priority       | `128`, если путь не требуется изменить  |

### Edge

Межкоммутаторный порт не должен быть edge:

```text
Edge: no
```

На нём ожидается получение BPDU.

### Point To Point

Для прямого полнодуплексного Ethernet-соединения:

```text
Point To Point: yes
```

Допустим также режим:

```text
Point To Point: auto
```

В режиме `auto` полнодуплексный Ethernet обычно определяется как point-to-point.

Не устанавливайте `yes`, если между устройствами находится:

* Ethernet-концентратор;
* общая полудуплексная среда;
* другое устройство, через которое на одном порту могут появляться несколько мостов.

### BPDU Guard

На межкоммутаторном порту:

```text
BPDU Guard: no
```

Иначе нормальный BPDU соседнего коммутатора отключит порт.

<a id="client-ports"></a>

## 5. Настройка клиентских портов

Клиентскими считаются порты, к которым подключены:

* компьютер;
* кассовый терминал;
* принтер;
* обычный сервер;
* другое конечное устройство без функции bridge.

Откройте:

> **Bridge → Ports → `<CLIENT_PORT>`**

Установите:

| Поле           | Значение              |
| -------------- | --------------------- |
| Edge           | `yes`                 |
| BPDU Guard     | `yes`                 |
| Point To Point | `auto`                |
| Priority       | оставить по умолчанию |
| Path Cost      | оставить по умолчанию |

### Edge

```text
Edge: yes
```

Порт может сразу перейти в `Forwarding`, не ожидая обычного процесса согласования RSTP.

Не включайте `Edge: yes` на:

* trunk;
* соединении между коммутаторами;
* порту к bridge;
* порту к точке доступа, если она может мостить другие сети.

### BPDU Guard

```text
BPDU Guard: yes
```

Если на клиентском порту появится BPDU, порт будет переведён в отключённое состояние.

Это защищает от:

* случайного подключения второго коммутатора;
* создания петли;
* попытки подменить Root Bridge.

### Проверка BPDU Guard

1. Настройте `Edge: yes` и `BPDU Guard: yes`.
2. Подключите к клиентскому порту другой коммутатор с включённым STP/RSTP.
3. После получения BPDU порт должен отключиться.

### Восстановление после BPDU Guard

1. Отключите посторонний коммутатор.

2. Откройте:

   > **Bridge → Ports**

3. Выберите заблокированный порт.

4. Установите:

   ```text
   Disabled: yes
   ```

5. Нажмите **Apply**.

6. Верните:

   ```text
   Disabled: no
   ```

7. Нажмите **Apply**.

Порт не восстанавливается автоматически, пока его вручную не выключить и не включить.

<a id="roles"></a>

## 6. Просмотр ролей и состояний

### 6.1. Проверка Root Bridge

На каждом MikroTik откройте:

> **Bridge → Bridge**

Откройте состояние `br-rstp`.

Проверьте:

* является ли устройство Root Bridge;
* Root Bridge ID;
* Root Path Cost;
* Root Port.

Ожидаемый результат:

```text
SW1: Root Bridge = yes
SW2: Root Bridge = no
SW3: Root Bridge = no
```

### 6.2. Проверка портов

Откройте:

> **Bridge → Ports**

Проверьте столбцы или состояние каждого порта:

* `Role`;
* `Forwarding`;
* `Learning`;
* `Path Cost`;
* `Root Path Cost`;
* `Designated Bridge`;
* `BPDU TX/RX`.

В ожидаемой топологии:

#### SW1

| Порт  | Роль       | Состояние  |
| ----- | ---------- | ---------- |
| к SW2 | Designated | Forwarding |
| к SW3 | Designated | Forwarding |

#### SW2

| Порт  | Роль       | Состояние  |
| ----- | ---------- | ---------- |
| к SW1 | Root       | Forwarding |
| к SW3 | Designated | Forwarding |

#### SW3

| Порт  | Роль      | Состояние  |
| ----- | --------- | ---------- |
| к SW1 | Root      | Forwarding |
| к SW2 | Alternate | Discarding |

Точное расположение Alternate Port может отличаться, если не были заданы приоритеты либо изменены стоимости.

### 6.3. Основные роли

| Роль            | Назначение                                    |
| --------------- | --------------------------------------------- |
| Root Port       | лучший путь данного коммутатора к Root Bridge |
| Designated Port | передающий порт выбранного сегмента           |
| Alternate Port  | резервный путь к Root Bridge                  |
| Backup Port     | резерв Designated Port в одном сегменте       |
| Disabled Port   | порт не участвует в передаче                  |

### 6.4. Состояния RSTP

| Состояние  | Передача пользовательского трафика |
| ---------- | ---------------------------------- |
| Discarding | нет                                |
| Learning   | ещё нет, но изучаются MAC          |
| Forwarding | да                                 |

## 7. Управление путём через Path Cost

`Path Cost` используется для выбора Root Port.

Меньшее значение предпочтительнее.

### 7.1. Пример задачи

Нужно заставить SW3 ходить к Root Bridge SW1 не напрямую, а через SW2:

```text
SW3 → SW2 → SW1
```

Вместо:

```text
SW3 → SW1
```

### 7.2. Настройка SW2

На SW2 откройте порт к SW1:

> **Bridge → Ports → ether5**

Установите:

```text
Path Cost: 10
```

### 7.3. Настройка SW3

Порт SW3 к SW1:

```text
Path Cost: 50
```

Порт SW3 к SW2:

```text
Path Cost: 10
```

Расчёт:

```text
Прямой путь SW3 → SW1:
50

Путь SW3 → SW2 → SW1:
10 + 10 = 20
```

SW3 должен выбрать порт к SW2 как Root Port.

Порт SW3 к SW1 станет резервным.

### 7.4. Возврат автоматической стоимости

Откройте порт и очистите вручную заданное значение либо выберите:

```text
Path Cost: auto
```

Название режима зависит от версии WebFig.

### Важное ограничение

Изменение Path Cost на Root Bridge не меняет его собственный Root Port: у Root Bridge его нет.

Path Cost в основном настраивают на не-root-коммутаторах, выбирающих путь к корню.

<a id="port-priority"></a>

## 8. Управление путём через Port Priority

Port Priority применяется, когда несколько путей имеют одинаковую стоимость.

Меньшее значение предпочтительнее.

Допустимый диапазон RouterOS:

```text
0–240
```

Обычно используются значения с шагом 16:

```text
0, 16, 32, 48, 64, 80, 96, 112, 128...
```

Значение по умолчанию:

```text
128
```

### 8.1. Пример с двумя параллельными линиями

```text
SW1 ether5 ═════ ether5 SW2
SW1 ether6 ═════ ether6 SW2
```

Обе линии имеют одинаковую стоимость.

Нужно сделать путь через `ether5` основным.

### 8.2. Настройка на SW1

Для `ether5`:

```text
Priority: 64
```

Для `ether6`:

```text
Priority: 128
```

SW2 должен предпочесть BPDU, пришедший от порта SW1 с меньшим Port ID.

Один порт SW2 станет Root Port, второй — Alternate Port.

### 8.3. Когда использовать Port Priority

Используйте его, если:

* Path Cost одинаков;
* upstream Bridge ID одинаков;
* нужно выбрать один из нескольких параллельных портов.

Для обычного изменения маршрута проще использовать Path Cost.

<a id="failover"></a>

## 9. Проверка отказоустойчивости

### 9.1. Подключение компьютеров

Подключите:

```text
PC-A → SW2 ether2
PC-B → SW3 ether2
```

Назначьте адреса одной подсети:

```text
PC-A: 192.168.10.10/24
PC-B: 192.168.10.20/24
```

Шлюз не нужен.

### 9.2. Исходный путь

При нормальной работе и ожидаемых ролях трафик проходит:

```text
PC-A
→ SW2
→ SW1
→ SW3
→ PC-B
```

Линия SW2–SW3 является резервной.

### 9.3. Обрыв активного кабеля

Запустите постоянный обмен между PC-A и PC-B.

Отключите:

```text
SW1 ↔ SW3
```

Ожидаемый результат:

1. SW3 перестаёт получать BPDU от SW1.
2. Порт SW3 к SW2 меняет роль `Alternate` на `Root`.
3. Порт переходит в `Forwarding`.
4. Новый путь:

   ```text
   PC-A
   → SW2
   → SW3
   → PC-B
   ```

Возможна кратковременная потеря нескольких пакетов.

### 9.4. Восстановление линии

Подключите кабель SW1–SW3 обратно.

RSTP снова выберет более предпочтительный путь:

```text
SW3 → SW1
```

Лишний путь через SW2–SW3 будет снова переведён в `Discarding`.

### 9.5. Отказ Root Bridge

Отключите SW1 либо оба его межкоммутаторных порта.

Ожидаемый результат:

1. SW2 имеет следующий наименьший Bridge Priority.
2. SW2 становится новым Root Bridge.
3. SW3 использует прямое соединение с SW2.
4. Оставшаяся сеть продолжает работать.

Устройства, подключённые только к выключенному SW1, будут недоступны.

### 9.6. Отказ промежуточного коммутатора

Если выключен SW2:

* хосты на SW2 становятся недоступны;
* SW1 и SW3 сохраняют прямое соединение;
* остальные работающие участки сети продолжают обмен.

RSTP защищает связность оставшейся топологии, но не заменяет двойное подключение конечного устройства.

<a id="rstp-vlan"></a>

## 10. RSTP совместно с VLAN

В VLAN-сети используйте один VLAN-aware bridge на каждом MikroTik.

Не создавайте отдельный bridge только ради RSTP.

### 10.1. Общая схема

```text
       trunk             trunk
SW1 ═════════ SW2 ═════════ SW3
 ╲                            ╱
  ═══════════ trunk ═════════
```

Через trunk могут проходить:

```text
VLAN 10
VLAN 20
VLAN 30
```

### 10.2. Настройка bridge

На каждом коммутаторе:

> **Bridge → Bridge → `<BRIDGE>`**

```text
Protocol Mode: rstp
VLAN Filtering: yes
```

VLAN Filtering настраивается по методичке [`02-vlan`](../02-vlan/).

### 10.3. Trunk-порты

На каждом trunk:

```text
Edge: no
Point To Point: yes
BPDU Guard: no
Ingress Filtering: yes
Frame Types: admit only VLAN tagged
```

В:

> **Bridge → VLANs**

trunk-порты должны быть `Tagged` для всех передаваемых VLAN.

### 10.4. Клиентские access-порты

На access-порту:

```text
Edge: yes
BPDU Guard: yes
PVID: номер клиентской VLAN
```

В таблице VLAN он указывается как `Untagged`.

### 10.5. Какое дерево используется

Обычный RSTP на bridge строит одно общее дерево для всех VLAN данного bridge.

Если один trunk заблокирован, он блокируется для пользовательского трафика всех VLAN этого RSTP-дерева.

Для разных путей у разных групп VLAN требуется MSTP. В базовом экзаменационном задании MSTP обычно не нужен.

### 10.6. BPDU и VLAN

Не создавайте VLAN-интерфейсы поверх физических trunk-портов и затем не добавляйте их обратно в другие bridge.

Для общей VLAN-топологии используйте:

```text
один bridge
+
Bridge VLAN Filtering
+
RSTP
```

<a id="rstp-lacp"></a>

## 11. RSTP совместно с LACP

LACP и RSTP решают разные задачи:

| Технология | Что резервирует                             |
| ---------- | ------------------------------------------- |
| LACP       | несколько линий между одной парой устройств |
| RSTP       | несколько путей через разные коммутаторы    |

### 11.1. Схема

```text
SW1 bond-lacp ═════ bond-lacp SW2
 \                              /
  \                            /
             SW3
```

### 11.2. Основное правило

В bridge добавляется:

```text
bond-lacp
```

Физические slave-порты:

```text
ether6
ether7
```

отдельно в bridge не добавляются.

### 11.3. Настройка bond как RSTP-порта

Откройте:

> **Bridge → Ports → bond-lacp**

Для соединения между коммутаторами:

| Поле           | Значение           |
| -------------- | ------------------ |
| Edge           | `no`               |
| Point To Point | `yes`              |
| BPDU Guard     | `no`               |
| Path Cost      | `auto` или вручную |

RSTP видит весь bond как один логический порт.

### 11.4. Реакция на отказ

Если пропал один slave-порт:

```text
LACP сохраняет bond через оставшиеся участники
```

RSTP не перестраивает дерево, пока весь логический bond остаётся рабочим.

Если пропали все участники bond:

```text
bond перестаёт работать
→ RSTP активирует альтернативный путь
```

### 11.5. Path Cost bond

Автоматическая стоимость bond не обязательно равна суммарной пропускной способности всех его участников.

Чтобы явно предпочесть LACP-канал, можно вручную назначить ему меньший `Path Cost`, чем резервному одиночному соединению.

<a id="rapier"></a>

## 12. STP/RSTP на Rapier

Настройка Rapier выполняется через AlliedWare CLI.

В примере используются:

```text
порт 2 — клиент
порт 5 — соединение с первым коммутатором
порт 6 — соединение со вторым коммутатором
```

Не изменяйте порты `23–26` без разрешения преподавателя.

### 12.1. Физическая схема

```text
             Rapier-A
          port5   port6
             /     \
            /       \
     port5             port5
      Rapier-B────────Rapier-C
            port6   port6
```

Сначала соедините сеть цепочкой и включите RSTP. Последний кабель подключайте после настройки.

---

## 12.2. Использование default STP instance

Для простой default VLAN можно использовать существующий экземпляр:

```text
default
```

### Rapier-A — основной Root

```text
set stp=default mode=rapid priority=4096
enable stp=default
```

### Rapier-B — резервный Root

```text
set stp=default mode=rapid priority=8192
enable stp=default
```

### Rapier-C

```text
set stp=default mode=rapid priority=12288
enable stp=default
```

---

## 12.3. Настройка межкоммутаторных портов

На каждом Rapier настройте используемые порты:

```text
set stp=default port=5,6 edgeport=no ptp=on
enable stp=default port=5,6
```

Значения:

```text
edgeport=no — порт ведёт к другому bridge
ptp=on      — прямое point-to-point соединение
```

---

## 12.4. Настройка клиентского порта

```text
set stp=default port=2 edgeport=yes ptp=auto
enable stp=default port=2
```

В старой версии AlliedWare отдельная функция BPDU Guard может отсутствовать.

Не блокируйте BPDU обычным фильтром на межкоммутаторных портах: это нарушит работу STP.

---

## 12.5. Просмотр состояния

Общая информация:

```text
show stp=default
```

Краткая информация:

```text
show stp=default summary
```

Состояние портов:

```text
show stp=default port=all
```

Подробное состояние RSTP:

```text
show stp=default port=all rstpstate
```

Счётчики:

```text
show stp=default counter
```

Проверьте:

* Root Bridge;
* Root Port;
* Designated Port;
* Alternate Port;
* Forwarding;
* Discarding.

---

## 12.6. Управление путём через Path Cost

Нужно сделать порт 5 основным, а порт 6 резервным:

```text
set stp=default port=5 pathcost=10
set stp=default port=6 pathcost=50
```

Меньшая стоимость предпочтительнее.

После изменения:

```text
show stp=default port=5,6 rstpstate
```

---

## 12.7. Управление путём через Port Priority

При одинаковом Path Cost:

```text
set stp=default port=5 portpriority=64
set stp=default port=6 portpriority=128
```

Меньшее значение предпочтительнее.

Port Priority используется как дополнительный критерий после стоимости пути.

---

## 12.8. Отдельный STP instance для VLAN

Если требуется отдельный экземпляр для VLAN `TEST`:

```text
create stp=LABSTP
```

Если VLAN уже относится к экземпляру `default`, сначала удалите её оттуда:

```text
delete stp=default vlan=TEST
```

Добавьте VLAN в новый экземпляр:

```text
add stp=LABSTP vlan=TEST
```

Настройте RSTP:

```text
set stp=LABSTP mode=rapid priority=4096
set stp=LABSTP port=5,6 edgeport=no ptp=on
set stp=LABSTP port=2 edgeport=yes ptp=auto
enable stp=LABSTP
```

На остальных Rapier задайте другие Bridge Priority:

```text
8192
12288
```

Все коммутаторы, передающие VLAN `TEST`, должны участвовать в соответствующем STP instance.

---

## 12.9. Проверка отказа канала Rapier

1. Подключите компьютеры к Rapier-B и Rapier-C.

2. Назначьте им адреса одной IP-подсети.

3. Организуйте постоянный обмен.

4. Отключите активный межкоммутаторный кабель.

5. Проверьте:

   ```text
   show stp=default port=all rstpstate
   ```

6. Резервный порт должен перейти в Forwarding.

7. Подключите кабель обратно.

8. Избыточный путь снова должен быть заблокирован.

Порт можно временно отключить программно:

```text
disable switch port=5
```

Вернуть:

```text
enable switch port=5
```

<a id="cleanup"></a>

## 13. Очистка конфигурации

### 13.1. MikroTik

Удаляйте настройки в таком порядке:

1. разомкните физическое кольцо;
2. отключите BPDU Guard на учебных портах, если он больше не нужен;
3. верните `Path Cost` к автоматическому значению;
4. верните `Port Priority` к `128`;
5. верните Bridge Priority к исходному значению, обычно `32768`;
6. удалите учебные порты из bridge;
7. удалите учебный bridge, если он больше не используется.

Если bridge используется VLAN и другими заданиями, не удаляйте его. Измените только параметры RSTP, созданные для эксперимента.

Чтобы полностью отключить STP на отдельном учебном bridge:

```text
Protocol Mode: none
```

В рабочей резервной L2-топологии отключать STP нельзя.

### 13.2. Rapier default instance

Разомкните физическое кольцо.

Верните приоритет:

```text
set stp=default priority=32768
```

Верните параметры портов:

```text
set stp=default port=5,6 pathcost=0 portpriority=128 edgeport=no ptp=auto
```

Если `pathcost=0` не поддерживается версией AlliedWare, используйте:

```text
reset stp=default
```

`reset` применяйте только в выделенной учебной конфигурации.

### 13.3. Удаление отдельного Rapier STP instance

Отключите экземпляр:

```text
disable stp=LABSTP
```

Удалите VLAN:

```text
delete stp=LABSTP vlan=TEST
```

Удалите экземпляр:

```text
destroy stp=LABSTP
```

При необходимости верните VLAN в default instance:

```text
add stp=default vlan=TEST
```

Не используйте `purge stp` на общем лабораторном коммутаторе: команда удаляет пользовательские STP instances.

## 14. Краткое соответствие параметров

| Задача                     | MikroTik WebFig                | Rapier                                 |
| -------------------------- | ------------------------------ | -------------------------------------- |
| Включить RSTP              | `Bridge → Protocol Mode=rstp`  | `set stp=... mode=rapid`               |
| Основной Root              | Bridge Priority `4096`         | `priority=4096`                        |
| Резервный Root             | Bridge Priority `8192`         | `priority=8192`                        |
| Root Port                  | выбирается по Path Cost        | выбирается по `pathcost`               |
| Основной путь              | меньший Path Cost              | меньший `pathcost`                     |
| Выбор при равной стоимости | Port Priority                  | `portpriority`                         |
| Клиентский порт            | `Edge=yes`                     | `edgeport=yes`                         |
| Межкоммутаторный порт      | `Edge=no`                      | `edgeport=no`                          |
| Point-to-point             | `Point To Point=yes`           | `ptp=on`                               |
| BPDU Guard                 | `BPDU Guard=yes`               | зависит от версии AlliedWare           |
| Просмотр Root              | состояние Bridge               | `show stp`                             |
| Просмотр ролей             | `Bridge → Ports`               | `show stp port=all rstpstate`          |
| Временно отключить линию   | Disable port или вынуть кабель | `disable switch port=...`              |
| VLAN trunk                 | tagged bridge port             | tagged VLAN port                       |
| LACP                       | bond является одним STP-портом | LACP trunk является логическим каналом |

## Минимальный алгоритм RSTP

```text
1. Нарисовать резервную L2-топологию.
2. Оставить последний кабель кольца отключённым.
3. Использовать один bridge на каждом коммутаторе.
4. Включить RSTP.
5. Назначить основной и резервный Root Bridge.
6. Межкоммутаторным портам задать Edge=no и Point-to-Point=yes.
7. Клиентским портам задать Edge=yes и BPDU Guard=yes.
8. Подключить последний кабель.
9. Найти Root, Root Port и Alternate Port.
10. Отключить активную линию и проверить переход резерва в Forwarding.
```
