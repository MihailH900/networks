# Подготовка оборудования с нуля

Эта методичка используется перед выполнением остальных практических заданий. Она описывает подготовку топологии, подключение к MikroTik и Rapier, настройку сетевых параметров Windows и Linux, а также безопасную очистку предыдущей конфигурации.

Все значения в угловых скобках нужно заменить параметрами своего задания:

* `<PORT>` — физический порт;
* `<IP>` — IP-адрес;
* `<PREFIX>` — длина префикса;
* `<GATEWAY>` — шлюз;
* `<DNS>` — DNS-сервер;
* `<VLAN_ID>` — номер VLAN.

## 1. Планирование сети

Перед подключением кабелей нарисуйте физическую топологию и составьте таблицы портов и адресов.

### 1.1. Физическая топология

Для каждого соединения укажите:

* устройство;
* номер порта;
* устройство на другом конце;
* номер порта на другом конце;
* назначение соединения.

Пример:

```text
PC-A ── ether2 MikroTik-1 ether5 ── ether5 MikroTik-2 ether2 ── PC-B
```

Для резервных линий показывайте каждый кабель отдельно:

```text
MikroTik-1 ether6 ═════════ ether6 MikroTik-2
MikroTik-1 ether7 ═════════ ether7 MikroTik-2
```

### 1.2. Таблица портов

| Устройство | Порт      | Подключение              | Назначение       |
| ---------- | --------- | ------------------------ | ---------------- |
| MikroTik-1 | `ether2`  | PC-A                     | клиентский порт  |
| MikroTik-1 | `ether5`  | MikroTik-2               | trunk            |
| MikroTik-1 | `ether6`  | MikroTik-2               | первый порт LACP |
| MikroTik-1 | `ether7`  | MikroTik-2               | второй порт LACP |
| MikroTik-1 | `ether10` | компьютер администратора | управление       |

Не используйте один физический порт одновременно для нескольких несовместимых назначений.

Например, один порт нельзя одновременно сделать:

* отдельным маршрутизируемым интерфейсом;
* участником bridge;
* slave-портом bonding;
* портом другого bridge.

### 1.3. IP-план

Для каждого L3-интерфейса запишите адрес, маску и шлюз.

| Устройство | Интерфейс             | IP-адрес           | Шлюз           |
| ---------- | --------------------- | ------------------ | -------------- |
| PC-A       | Ethernet              | `192.168.10.10/24` | `192.168.10.1` |
| MikroTik-1 | `ether2` или `vlan10` | `192.168.10.1/24`  | —              |
| MikroTik-1 | `ether3`              | `10.0.12.1/30`     | —              |
| MikroTik-2 | `ether3`              | `10.0.12.2/30`     | —              |

Проверяйте, что:

* IP-адрес не является адресом сети;
* IP-адрес не является broadcast;
* два устройства не используют один IP;
* шлюз клиента находится в его подсети;
* транзитные интерфейсы находятся в одной подсети.

### 1.4. VLAN-план

Для каждой VLAN укажите назначение, подсеть и порты.

| VLAN ID | Назначение | Подсеть           | Access-порты | Trunk-порты |
| ------: | ---------- | ----------------- | ------------ | ----------- |
|      10 | Services   | `192.168.10.0/24` | `ether2`     | `ether5`    |
|      20 | Staff      | `192.168.20.0/24` | `ether3`     | `ether5`    |
|      30 | Guest      | `192.168.30.0/24` | `ether4`     | `ether5`    |
|     999 | Quarantine | `192.168.99.0/24` | динамически  | `ether5`    |

Используйте одинаковые VLAN ID на всех устройствах, через которые должна проходить одна VLAN.

Рекомендуемые учебные номера:

```text
10–99    — пользовательские и серверные VLAN
100      — management
999      — quarantine
```

Номера можно заменить на указанные в задании.

### 1.5. Management-порт

По возможности выделите отдельный физический порт только для управления.

Пример:

```text
ether10 — management
```

Этот порт:

* не включается в экспериментальный bridge;
* не становится trunk;
* не добавляется в bonding;
* не используется для клиентского трафика;
* получает отдельный management-IP.

Пример:

```text
MikroTik management: 192.168.100.1/24
Компьютер администратора: 192.168.100.10/24
```

Если отдельный management-порт отсутствует, доступ необходимо подготовить через management VLAN до включения VLAN Filtering.

### 1.6. Зарезервированные порты

Перед началом уточните у преподавателя:

* какие порты используются для управления;
* какие порты соединяют оборудование лаборатории;
* разрешено ли изменять существующий bridge;
* разрешено ли сбрасывать конфигурацию;
* разрешено ли менять management-IP.

Не изменяйте неизвестные подключения и зарезервированные порты.

## 2. Подключение и подготовка MikroTik

Основная настройка MikroTik выполняется через WebFig.

### 2.1. Физическое подключение

1. Подключите компьютер одним Ethernet-кабелем к выбранному management-порту MikroTik.
2. Не подключайте остальные учебные кабели до проверки доступа.
3. Отключите Wi-Fi и другие неиспользуемые сетевые адаптеры компьютера.

### 2.2. Определение management-IP

Возможны три ситуации.

#### Адрес известен

Откройте в браузере:

```text
http://<MANAGEMENT_IP>
```

или, если настроен HTTPS:

```text
https://<MANAGEMENT_IP>
```

#### Адрес указан преподавателем

Назначьте компьютеру временный адрес из той же подсети.

Пример:

```text
MikroTik: 192.168.100.1/24
Компьютер: 192.168.100.10/24
```

#### Адрес неизвестен

Для первоначального обнаружения можно один раз использовать WinBox:

1. открыть вкладку **Neighbors**;
2. найти устройство по MAC-адресу;
3. посмотреть его IP;
4. дальнейшую настройку выполнять через WebFig.

Не сбрасывайте MikroTik только из-за неизвестного IP.

### 2.3. Вход в WebFig

1. Откройте браузер.
2. Введите management-IP MikroTik.
3. Введите логин и пароль.
4. Убедитесь, что открыта конфигурация нужного устройства.

### 2.4. Определение версии RouterOS

Откройте:

> **System → Resources**

Запишите значение **Version**.

Названия отдельных полей могут различаться между версиями RouterOS, но основные разделы сохраняются.

### 2.5. Просмотр существующей конфигурации

Перед изменениями откройте следующие разделы.

#### Физические и логические интерфейсы

> **Interfaces**

Проверьте:

* какие порты находятся в состоянии Running;
* какие VLAN-интерфейсы уже существуют;
* есть ли bonding-интерфейсы;
* какие интерфейсы отключены.

#### Bridge

> **Bridge → Bridge**

Проверьте:

* список bridge;
* значение `Protocol Mode`;
* включён ли `VLAN Filtering`.

#### Порты bridge

> **Bridge → Ports**

Проверьте:

* какой физический порт входит в какой bridge;
* PVID;
* edge;
* BPDU Guard;
* ingress filtering.

#### Таблица VLAN

> **Bridge → VLANs**

Проверьте:

* VLAN ID;
* tagged-порты;
* untagged-порты;
* динамические записи.

#### IP-адреса

> **IP → Addresses**

Проверьте:

* адрес;
* маску;
* интерфейс;
* management-IP.

#### Маршруты

> **IP → Routes**

Проверьте наличие маршрута к management-сети и маршрута по умолчанию.

### 2.6. Назначение management-IP

Если management-порт не входит в bridge:

> **IP → Addresses → Add New**

Укажите:

| Поле      | Значение                   |
| --------- | -------------------------- |
| Address   | `<MANAGEMENT_IP>/<PREFIX>` |
| Interface | `<MANAGEMENT_PORT>`        |

Пример:

```text
Address: 192.168.100.1/24
Interface: ether10
```

Если management-порт входит в отдельный management bridge, IP назначается bridge, а не физическому порту.

Пример:

```text
Address: 192.168.100.1/24
Interface: br-management
```

Не назначайте один и тот же IP одновременно bridge и его физическому порту.

### 2.7. Сохранение доступа при включении VLAN Filtering

Самый безопасный вариант:

```text
management-порт не входит в экспериментальный bridge
```

Если управление должно проходить через VLAN:

1. создайте management VLAN;
2. добавьте management-порт как access или trunk;
3. добавьте bridge в соответствующую VLAN как tagged CPU-порт;
4. создайте VLAN-интерфейс поверх bridge;
5. назначьте management-IP VLAN-интерфейсу;
6. проверьте вход через новый адрес;
7. только после этого включите `VLAN Filtering`.

Не включайте VLAN Filtering, пока management-доступ не подготовлен полностью.

## 3. Подключение и подготовка Rapier

Rapier используется как альтернативный вариант оборудования. Часть функций может быть доступна только через AlliedWare CLI.

### 3.1. Физическое подключение

1. Подключите компьютер к разрешённому пользовательскому или management-порту.
2. Не используйте зарезервированные порты без разрешения преподавателя.
3. Отключите лишние сетевые адаптеры компьютера.

### 3.2. Определение адреса управления

Management-IP может быть:

* указан преподавателем;
* указан в лабораторной схеме;
* сохранён от предыдущей настройки;
* доступен через консоль устройства.

Назначьте компьютеру адрес из той же подсети.

Пример:

```text
Rapier: 192.168.100.2/24
Компьютер: 192.168.100.10/24
```

### 3.3. Вход в веб-интерфейс

Откройте:

```text
http://<RAPIER_IP>
```

Введите учётные данные.

Через веб-интерфейс выполняйте доступные базовые настройки:

* IP управления;
* состояние портов;
* VLAN;
* скорость и дуплекс;
* базовую коммутацию.

### 3.4. Когда нужен CLI

Консоль, Telnet или SSH могут потребоваться для:

* GVRP;
* LACP;
* расширенного STP;
* Port Security;
* RADIUS и 802.1X;
* функций, отсутствующих в веб-интерфейсе конкретной модели.

Используйте CLI только в соответствующем разделе основной методички.

### 3.5. Правила безопасного изменения

Перед изменениями:

1. определите порт, через который выполняется управление;
2. определите management VLAN;
3. не удаляйте management VLAN;
4. не переводите management-порт в другую VLAN без подготовленного запасного доступа;
5. не отключайте неизвестные trunk-порты;
6. не сохраняйте конфигурацию поверх общей лабораторной настройки без разрешения.

## 4. Настройка Windows-клиента и Windows Server

### 4.1. Выбор сетевого адаптера

Откройте список сетевых подключений:

> **Панель управления → Сеть и Интернет → Сетевые подключения**

На Windows Server 2003 или Windows XP:

> **Панель управления → Сетевые подключения**

Определите нужный Ethernet-адаптер по состоянию подключения.

Отключите:

* Wi-Fi;
* VPN;
* виртуальные адаптеры;
* неиспользуемые Ethernet-интерфейсы.

Не отключайте адаптер, через который выполняется удалённое управление сервером.

### 4.2. Статический IPv4

1. Откройте свойства нужного Ethernet-адаптера.
2. Выберите **Internet Protocol Version 4 (TCP/IPv4)**.
3. Откройте **Свойства**.
4. Выберите ручную настройку.
5. Укажите:

```text
IP-адрес:      <IP>
Маска:         <MASK>
Основной шлюз: <GATEWAY>
DNS-сервер:    <DNS>
```

Пример:

```text
IP:      192.168.10.10
Маска:   255.255.255.0
Шлюз:    192.168.10.1
DNS:     192.168.10.2
```

Для локального теста без выхода в другие сети шлюз и DNS можно временно не указывать.

### 4.3. Автоматическое получение адреса

В свойствах IPv4 выберите:

* **Получить IP-адрес автоматически**;
* **Получить адрес DNS-сервера автоматически**.

После изменения отключите и снова включите сетевое подключение либо переподключите кабель.

### 4.4. Определение MAC-адреса

Откройте:

> **Состояние адаптера → Сведения**

Найдите поле:

```text
Физический адрес
```

Запишите MAC в формате, который требуется RADIUS или Port Security.

Пример:

```text
00-11-22-33-44-55
```

или:

```text
00:11:22:33:44:55
```

### 4.5. Службы Windows

Откройте:

> **Панель управления → Администрирование → Службы**

Для дальнейших заданий могут потребоваться:

* DHCP Server;
* Internet Authentication Service;
* Network Policy Server;
* Routing and Remote Access;
* Wired AutoConfig.

Конкретные службы включаются в соответствующих методичках.

### 4.6. Оснастки MMC

Откройте:

> **Панель управления → Администрирование**

Используйте нужную оснастку:

* DHCP;
* Internet Authentication Service;
* Network Policy Server;
* Routing and Remote Access;
* Computer Management.

### 4.7. Windows Firewall

Не отключайте firewall полностью без необходимости.

Разрешайте только требуемые службы и направления:

* DHCP;
* RADIUS;
* ICMP;
* DNS;
* веб-сервер;
* управление.

На время изолированной лабораторной работы firewall можно временно отключить только с разрешения преподавателя. После завершения его необходимо включить обратно.

## 5. Настройка Linux-клиента

Основной способ — графические настройки сети.

### 5.1. Выбор физического интерфейса

Откройте:

> **Настройки → Сеть → Проводное подключение**

Нужный интерфейс обычно определяется по изменению состояния при подключении и отключении кабеля.

Отключите:

* Wi-Fi;
* VPN;
* ненужные проводные подключения;
* виртуальные адаптеры.

### 5.2. Статический IPv4

Откройте:

> **Проводное подключение → Настройки → IPv4**

Выберите:

```text
Метод: Вручную
```

Укажите:

```text
Адрес:  <IP>
Маска:  <PREFIX> или <MASK>
Шлюз:   <GATEWAY>
DNS:    <DNS>
```

Пример:

```text
Адрес:  192.168.10.10
Маска:  255.255.255.0
Шлюз:   192.168.10.1
DNS:    192.168.10.2
```

Сохраните настройки и переподключите интерфейс.

### 5.3. DHCP

В разделе IPv4 выберите:

```text
Метод: Автоматически (DHCP)
```

Сохраните настройки и переподключите интерфейс.

### 5.4. Минимальные команды

Команды применяются только для определения имени интерфейса и проверки назначенных параметров.

Список интерфейсов:

```bash
ip -br link
```

Назначенные адреса:

```bash
ip -br address
```

Маршруты:

```bash
ip route
```

MAC конкретного интерфейса:

```bash
ip link show <INTERFACE>
```

Пример имени физического интерфейса:

```text
enp2s0
```

<a id="cleanup"></a>

## 6. Очистка предыдущего задания

Перед удалением конфигурации запишите:

* через какой порт открыт WebFig;
* на каком интерфейсе находится management-IP;
* какие порты зарезервированы;
* какие настройки существовали до начала работы.

### 6.1. Общий порядок очистки MikroTik

Удаляйте конфигурацию от зависимых объектов к базовым.

Рекомендуемый порядок:

1. отключить созданные серверы и relay;
2. удалить правила firewall и очереди, созданные для задания;
3. удалить IP-адреса с учебных VLAN и интерфейсов;
4. удалить VLAN-интерфейсы;
5. удалить записи **Bridge → VLANs**;
6. удалить bond из bridge;
7. удалить bonding-интерфейс;
8. только после удаления bond повторно использовать его slave-порты;
9. удалить учебные порты из bridge;
10. удалить сам учебный bridge, если он больше не нужен.

Не удаляйте:

* management-IP;
* management bridge;
* порт, через который открыт WebFig;
* неизвестные системные bridge;
* настройки других участников.

### 6.2. Освобождение порта из bridge

Откройте:

> **Bridge → Ports**

Найдите физический интерфейс и удалите или отключите его запись.

После этого порт можно использовать как:

* отдельный L3-интерфейс;
* участник другого bridge;
* slave bonding;
* management-порт.

Один физический порт не должен одновременно входить в два bridge.

### 6.3. Удаление VLAN

Сначала удалите зависимые объекты:

* DHCP Server;
* DHCP Relay;
* IP-адрес;
* VLAN-интерфейс.

Затем:

1. откройте **Bridge → VLANs**;
2. удалите VLAN из tagged и untagged membership;
3. откройте **Bridge → Ports**;
4. верните PVID порта к требуемому значению;
5. при необходимости отключите VLAN Filtering.

Не отключайте VLAN Filtering на рабочем общем bridge без необходимости.

### 6.4. Разбор bonding

1. Удалите bond из **Bridge → Ports**.
2. Удалите IP и VLAN-интерфейсы с bond.
3. Откройте **Interfaces → Bonding**.
4. Удалите bonding-интерфейс.
5. Убедитесь, что физические slave-порты снова появились как самостоятельные интерфейсы.
6. Только после этого добавляйте их в bridge или назначайте им IP.

### 6.5. Очистка Rapier

Удаляйте настройки в обратном порядке:

1. аутентификация и ограничения порта;
2. LACP или статический агрегат;
3. динамическое членство GVRP;
4. tagged и untagged membership;
5. созданные VLAN;
6. IP-адреса учебных интерфейсов.

Не удаляйте management VLAN и IP управления.

### 6.6. Возврат Windows и Linux

После завершения задания:

* включите ранее отключённые адаптеры;
* верните DHCP, если статический адрес больше не нужен;
* удалите временные шлюзы и DNS;
* включите firewall;
* остановите временно созданные серверные службы, если они больше не используются.

<a id="final-check"></a>

## 7. Минимальная проверка готовности

Перед переходом к основной методичке убедитесь, что:

* физические кабели подключены согласно таблице портов;
